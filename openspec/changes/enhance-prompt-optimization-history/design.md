## 上下文
现有提示词优化链路已具备：
- 两阶段生成问题分析与优化后提示词
- 人工微调优化后提示词
- 基于优化后提示词创建对比任务
- 任务级指标对比
- 一键替换当前任务 Prompt

但当前实现把一个任务的优化结果建模为“单条最新记录”，这与真实使用方式不匹配。Prompt 优化本质是多轮迭代过程，系统应当保留每次优化的独立版本，避免丢失历史上下文。

此外，对比卡片如果只展示数值，会让用户很难快速判断哪些指标变好了、哪些指标变差了。尤其在指标较多时，视觉辅助很重要。

## 目标
- 保留提示词优化的历史版本
- 让每次重新生成都成为新的优化版本
- 支持在任务详情页切换查看不同版本
- 为任务级对比增加提升/下降的颜色与箭头反馈

## 非目标
- 不做复杂的 Prompt 分支管理
- 不引入版本合并、多人协作审批
- 不在本次变更中提供逐样本对比的升降可视化

## 决策
### 1. 优化结果按版本持久化
每次重新生成提示词优化结果时，系统必须创建一条新的优化版本记录，而不是覆盖旧版本。

### 2. 当前任务存在“最新版本”概念
任务详情页默认展示最新优化版本，但用户可以切换查看历史版本。

### 3. 历史版本保持独立上下文
每个版本都应独立保存：
- 问题分析摘要
- 问题点
- 优化策略
- 模型生成的优化后提示词
- 人工微调后的提示词
- 对比任务引用

### 4. 对比卡片增加视觉反馈
对比卡片中的每个任务级指标都要显示：
- 当前值
- 与基线相比是提升还是下降
- 对应颜色与箭头

建议口径：
- 召回率/精确率/覆盖率/空样本通过率：高于基线为提升
- 不可评分数：低于基线为提升

## 风险 / 权衡
- 历史版本会让数据模型从“一对一”变成“一对多”，需要兼顾当前最新版本的读取效率
- 替换当前任务 Prompt 时需要明确“应用的是哪一个版本”的编辑结果
- 如果历史版本很多，前端需要简洁展示，避免版本列表过长影响可用性

## 建议实现方向
- 将优化结果从任务的一对一记录调整为一对多版本记录
- 在记录中增加版本号或创建时间排序能力
- 新增按任务获取优化版本列表的接口
- 当前“获取优化结果”接口可默认返回最新版本，或支持通过版本 ID 查询
- 前端在优化结果区域新增：
  - 历史版本切换器
  - 指标升降视觉反馈
